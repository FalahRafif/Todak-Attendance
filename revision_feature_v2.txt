2. PRIORITAS TINGGI - WAJIB DIREVISI BERIKUTNYA
====================================================================

--------------------------------------------------------------------
2.1 Tambahkan DB::transaction untuk HRD approve leave
--------------------------------------------------------------------

Masalah:

Proses HRD approve leave melakukan beberapa operasi multi-table:

- update leave_requests
- insert leave_request_details
- create/update attendances
- update leave_balances untuk annual_leave

Saat ini proses tersebut belum dipastikan dibungkus DB::transaction. Jika salah satu proses gagal di tengah, data bisa tidak konsisten.

Instruksi revisi:

1. Bungkus seluruh proses approve leave di HrdService dengan DB::transaction.
2. Pastikan semua operasi approve berada dalam satu transaction.
3. Jika salah satu proses gagal, rollback semua perubahan.
4. Pastikan validasi status pending dilakukan di dalam transaction.
5. Untuk annual_leave, lock leave_balances sebelum mengurangi saldo.

Contoh pola:

DB::transaction(function () use ($leaveRequestId, $payload) {
    // ambil leave request
    // validasi status pending
    // validasi overlap
    // jika annual leave, ambil leave balance dengan lockForUpdate()
    // update leave request menjadi approved
    // insert leave_request_details per tanggal
    // create/update attendances per tanggal
    // kurangi leave balance
    // insert activity log / approval log jika sudah tersedia
});

Acceptance criteria:

- Jika approve leave berhasil, semua tabel terkait berubah konsisten.
- Jika gagal di tengah, tidak ada data parsial yang tersimpan.
- Annual leave tidak bisa mengurangi saldo dua kali akibat race condition.
- Approve leave yang sudah bukan pending harus ditolak.

--------------------------------------------------------------------
2.2 Tambahkan DB::transaction untuk HRD approve correction
--------------------------------------------------------------------

Masalah:

Approve correction melakukan beberapa operasi:

- update attendance_correction_requests
- create/update attendances
- insert attendance_logs
- hitung ulang late_minutes, total_work_minutes, early_leave_minutes, status present/late

Belum dipastikan transaction-safe.

Instruksi revisi:

1. Bungkus proses approve correction dengan DB::transaction.
2. Validasi status correction harus pending di dalam transaction.
3. Jika correction sudah approved/rejected/cancelled, route langsung harus ditolak.
4. Insert attendance_logs hanya dilakukan jika update attendance berhasil.

Acceptance criteria:

- Correction pending bisa approved satu kali saja.
- Correction yang sudah approved tidak bisa diproses ulang walau route ditembak langsung.
- Jika update attendance gagal, correction tidak berubah menjadi approved.
- Attendance log tidak tercipta jika approval gagal.

--------------------------------------------------------------------
2.3 Tambahkan lockForUpdate untuk leave_balances
--------------------------------------------------------------------

Masalah:

Leave balance reduce belum idempotent dan rawan race condition jika dua approval annual leave berjalan bersamaan.

Instruksi revisi:

1. Saat approve annual_leave, ambil row leave_balances dengan lockForUpdate.
2. Validasi remaining_quota di dalam lock.
3. Kurangi saldo hanya setelah validasi berhasil.
4. Jangan kurangi saldo jika leave request sudah bukan pending.

Contoh pola:

$balance = LeaveBalance::where('employee_id', $employeeId)
    ->where('year', $year)
    ->where('leave_type_id', $annualLeaveTypeId)
    ->lockForUpdate()
    ->first();

Acceptance criteria:

- Dua approval bersamaan tidak menyebabkan saldo minus.
- Saldo hanya berkurang sekali untuk satu leave request.
- Jika saldo tidak cukup, proses approve gagal dengan pesan bisnis yang jelas.

2.6 Perbaiki monthly summary agar UUID tidak berubah saat generate ulang
--------------------------------------------------------------------

Masalah:

Generate monthly summary masih memakai uuid di update payload pada updateOrCreate. Ini bisa membuat UUID existing berubah setiap generate ulang.

Instruksi revisi:

1. Jangan masukkan uuid ke update payload updateOrCreate.
2. UUID hanya dibuat saat data summary baru dibuat.
3. Gunakan key unik: employee_id + year + month.
4. Pastikan unique constraint employee_id, year, month sudah ada.

Pola yang benar:

$summary = AttendanceMonthlySummary::firstOrNew([
    'employee_id' => $employeeId,
    'year' => $year,
    'month' => $month,
]);

if (!$summary->exists) {
    $summary->uuid = (string) Str::uuid();
}

$summary->fill([
    'total_work_days' => ...,
    'total_present' => ...,
    // dst
]);

$summary->save();

Acceptance criteria:

- Generate summary pertama membuat UUID.
- Generate ulang tidak mengubah UUID yang sudah ada.
- Summary tetap update angka terbaru.

--------------------------------------------------------------------

Masalah:

Holiday sudah tampil di calendar dan HRD belum check-in sudah exclude holiday, tetapi Employee check-in belum memblokir atau menandai khusus hari libur.

Keputusan yang dibutuhkan:

2. Jika tetap absen, simpan attendance dengan flag/catatan holiday attendance.
3. Jika belum ada lembur/overtime, tampilkan warning: "Hari ini libur. Absensi tidak diwajibkan."

Implementasi minimal:

--------------------------------------------------------------------
9.2 Leave lintas tahun
--------------------------------------------------------------------

Masalah:

Validasi kuota annual leave saat ini memakai tahun start_date. Jika cuti lintas tahun, saldo harus dihitung terpisah.

Contoh:

Cuti 30 Desember 2026 sampai 2 Januari 2027:

- Tahun 2026: 2 hari
- Tahun 2027: 2 hari

Keputusan yang dibutuhkan:

1. Apakah cuti lintas tahun diizinkan?
2. Jika diizinkan, saldo harus dihitung per tahun.
3. Jika tidak diizinkan, sistem harus menolak pengajuan lintas tahun.

Saran MVP:

- Tolak cuti annual_leave lintas tahun terlebih dahulu.
