================================================================================
A. RINGKASAN STATUS TERBARU
================================================================================

Progress sistem sudah meningkat signifikan. Aplikasi sudah tidak hanya mencakup absensi dasar, tetapi sudah memiliki modul HRD yang cukup lengkap seperti:

- Auth login/logout dan role-based access Admin, HRD, Employee.
- Branding UI login dan APP_NAME sudah diganti ke Todak Attendace.
- Admin CRUD master utama: department, position, work location, shift, holiday, employee.
- Admin Parameter Aplikasi untuk kuota cuti tahunan global.
- Employee dashboard, attendance today, check-in, check-out, history, detail history, calendar, leave request, correction request, profile.
- HRD dashboard, attendance monitoring, attendance map, activity calendar, not checked-in, incomplete, late, outside radius, approval leave, approval correction, report, leave balance.
- Attendance detail HRD dan Employee sudah menampilkan foto check-in/check-out via signed preview URL.
- Attendance detail HRD dan Employee sudah menampilkan map, lokasi kantor, radius, marker employee, dan garis pembanding.

Estimasi status:
- MVP absensi dasar: 85-90% selesai.
- Business rule HR: 80-85% selesai.
- HRD monitoring: 75-80% selesai.
- Report/export: 55-65% selesai.
- Audit & compliance: 35-45% selesai.
- Admin system management: 65-70% selesai.

Kesimpulan:
Sistem sudah layak masuk QA browser end-to-end, tetapi belum boleh dianggap selesai penuh untuk production yang rapi.

2. EMPLOYEE ATTENDANCE
Flow absensi employee sudah cukup matang:
- Check-in.
- Check-out.
- Kamera/selfie wajib.
- GPS wajib.
- Jarak ke work location dihitung.
- Outside radius tetap boleh absen.
- Outside radius wajib work_mode_id + note.
- Work mode office ditolak jika outside radius.
- Duplicate check-in/check-out ditolak.
- Late status otomatis.
- Early leave dihitung.
- Attendance log check_in/check_out dibuat.
- Detail attendance menampilkan foto dan map.

3. LEAVE REQUEST
Leave request sudah jauh lebih baik:
- Validasi overlap pending/approved.
- Validasi saldo annual leave.
- Annual leave lintas tahun ditolak.
- Cancel menjadi status cancelled, bukan delete row.
- HRD approve membuat attendance leave/sick/permission.
- HRD approve membuat leave_request_details.
- HRD approve mengurangi leave_balances dengan row lock.
- Approve dibungkus DB::transaction.

4. ATTENDANCE CORRECTION
Correction sudah lebih matang:
- Employee bisa mengajukan koreksi.
- Validasi duplikasi pending per tanggal.
- Cancel menjadi status cancelled.
- Approve/reject hanya untuk status pending.
- Approve dibungkus DB::transaction.
- Approve create/update attendance.
- Recalculate late_minutes, total_work_minutes, early_leave_minutes, dan status present/late.

5. HRD MONITORING
HRD sudah memiliki:
- Dashboard.
- Monitoring absensi.
- Peta absensi.
- Activity calendar.
- Belum check-in.
- Incomplete/belum check-out.
- Late.
- Outside radius.
- Approval leave.
- Approval correction.
- Report harian/bulanan.
- Leave balance management.

6. LEAVE BALANCE HRD
Leave balance sudah cukup operasional:
- Generate missing balance.
- Adjust tambahan quota/reward.
- Set total quota.
- Menggunakan setting Admin DEFAULT_ANNUAL_LEAVE_QUOTA.

1. UPLOAD ATTACHMENT UNTUK LEAVE DAN CORRECTION BELUM ADA

Masalah:
- Tabel leave_requests dan attendance_correction_requests sudah punya attachment_id.
- Form Employee belum menyediakan upload attachment pendukung.
- HRD detail belum menampilkan attachment leave/correction.

Dampak:
- Pengajuan sakit tidak bisa melampirkan surat dokter.
- Pengajuan izin tidak bisa melampirkan bukti.
- Koreksi absensi tidak bisa melampirkan bukti meeting/surat tugas/screenshot.

Yang harus dibuat:
- Tambahkan upload attachment pada form Employee leave request.
- Tambahkan upload attachment pada form Employee attendance correction.
- Simpan file melalui attachment service.
- Simpan attachment_id ke leave_requests atau attendance_correction_requests.
- Tampilkan attachment di HRD leave request detail.
- Tampilkan attachment di HRD correction detail.
- Gunakan signed preview/download URL.
- Validasi file extension, mime type, dan ukuran file.
- Untuk MVP, attachment bisa optional.
- Untuk sick_leave, nanti bisa dibuat wajib melalui setting REQUIRE_SICK_ATTACHMENT.

Acceptance Criteria:
- Employee bisa submit leave dengan attachment optional.
- Employee bisa submit correction dengan attachment optional.
- HRD bisa melihat/download attachment melalui signed URL.
- Direct public URL tidak ditampilkan di UI.
- File tidak bisa diakses oleh user yang tidak berwenang.

4. APPROVALS TABLE BELUM DIPAKAI

Masalah:
- Tabel approvals sudah ada.
- Approval leave/correction/outside radius masih langsung update tabel target.
- Audit approval belum terpusat.

Yang harus dibuat:
- Saat HRD approve/reject leave, insert row ke approvals.
- Saat HRD approve/reject attendance correction, insert row ke approvals.
- Saat HRD approve/reject outside radius, insert row ke approvals atau minimal siapkan kompatibilitasnya.

Contoh data approvals:
- approvable_type = leave_request / attendance_correction / outside_radius_attendance.
- approvable_id = ID data target.
- requested_by = user_id employee.
- approved_by = user_id HRD.
- status_id = approved/rejected.
- note = approval/rejection note.
- approved_at = now().
- approval_level = 1.

Acceptance Criteria:
- Setiap approval/rejection leave membuat row approvals.
- Setiap approval/rejection correction membuat row approvals.
- Data approvals bisa dipakai untuk audit tanpa harus baca banyak tabel target.
- Flow existing tetap update tabel target seperti sebelumnya.

5. OUTSIDE RADIUS REJECT BELUM PUNYA STATUS REVIEW YANG JELAS

Masalah:
- Outside radius sudah bisa direview.
- Namun reject outside radius belum punya status review yang kuat.
- Report belum bisa membedakan outside radius approved vs rejected secara ideal.

Rekomendasi:
Tambahkan field atau gunakan approvals table.

Opsi A - Tambah field di attendances:
- outside_radius_review_status_id.
- outside_radius_reviewed_by.
- outside_radius_reviewed_at.
- outside_radius_review_note.

Status:
- pending_review.
- approved.
- rejected.

Opsi B - Gunakan approvals table:
- approvable_type = outside_radius_attendance.
- approvable_id = attendances.id.
- status_id = approved/rejected.

Flow ideal:
- Jika outside radius:
  - is_need_approval = true.
  - outside_radius_review_status = pending_review.
- HRD approve:
  - outside_radius_review_status = approved.
  - is_need_approval = false.
- HRD reject:
  - outside_radius_review_status = rejected.
  - is_need_approval = false atau tetap true sesuai kebijakan.
  - HRD note wajib.

Saran MVP:
- Jangan otomatis mengubah attendance menjadi absent.
- Simpan sebagai outside radius rejected.
- Efek payroll/penilaian bisa diputuskan belakangan.

Acceptance Criteria:
- HRD bisa approve outside radius.
- HRD bisa reject outside radius dengan note.
- Report bisa membedakan pending/approved/rejected outside radius.
- Status review terlihat di detail attendance.

6. SCHEDULE MANAGEMENT BELUM ADA

Masalah:
- employee_schedules sudah dipakai untuk exclude day off pada HRD belum check-in.
- Namun belum ada create/update/delete schedule.
- Employee attendance masih memakai shift utama employee, belum override dari employee_schedules.

Flow shift ideal:
1. Cek employee_schedules berdasarkan employee_id dan tanggal hari ini.
2. Jika ada dan is_day_off = true:
   - employee tidak wajib absen.
3. Jika ada dan shift_id terisi:
   - pakai shift dari employee_schedules.
4. Jika tidak ada schedule:
   - fallback ke employees.shift_id.

Fitur schedule management:
- List schedule per bulan.
- Create schedule.
- Edit schedule.
- Delete schedule.
- Set day off.
- Bulk generate schedule.
- Filter by employee/department/month.
- Validasi unique employee_id + schedule_date.

Acceptance Criteria:
- Admin/HRD bisa membuat jadwal harian employee.
- Admin/HRD bisa set day off.
- Employee dashboard/attendance memakai shift dari employee_schedules jika ada.
- HRD not checked-in memakai schedule harian secara konsisten.
- Jika is_day_off true, Employee diberi info bahwa hari tersebut day off.

F. FLOW YANG MASIH PERLU KEPUTUSAN BISNIS
================================================================================

1. ABSENSI SAAT HARI LIBUR

Kondisi saat ini:
- Holiday sudah tampil di calendar.
- HRD not checked-in sudah exclude holiday.
- Employee check-in belum memblokir/menandai khusus saat holiday.

Perlu keputusan:
- Apakah employee boleh absen saat holiday?

Rekomendasi MVP:
- Hari libur tidak wajib absen.
- Tampilkan warning: "Hari ini libur. Absensi tidak diwajibkan."
- Jika belum ada lembur/overtime, tombol absen bisa disabled saat holiday.
- Jika tetap ingin boleh absen, simpan sebagai holiday attendance dengan flag/catatan.

2. DAMPAK REJECT OUTSIDE RADIUS

Perlu keputusan:
Jika HRD reject outside radius, apakah:
- Attendance tetap present tetapi review_status = rejected.
- Attendance berubah menjadi absent.
- Attendance tetap pending dan employee wajib submit correction.

Rekomendasi MVP:
- Jangan otomatis ubah ke absent.
- Simpan review_status = rejected.
- HRD note wajib.
- Report tampilkan "Outside Radius Rejected".

3. APPROVED LEAVE YANG DIBATALKAN

Kondisi:
- Belum ada mekanisme restore saldo jika leave approved dibatalkan pada future flow.

Jika nanti ada fitur cancel approved leave, wajib:
- Restore leave balance.
- Update attendance status.
- Update/delete leave_request_details.
- Catat activity log.
- Catat approval/audit trail.
