# Revision Feature KlikAbsen

Tanggal revisi: 2026-06-09
Tujuan dokumen: menjadi instruksi detail untuk AI agent/internal developer agar memperbaiki fitur, flow bisnis, validasi, report, dan audit trail KlikAbsen berdasarkan hasil audit akhir.

Dokumen ini fokus pada hal yang masih kurang, flow yang belum semestinya, dan prioritas pengerjaan. Jangan mengubah struktur utama project, jangan revive modul Etherno, dan tetap ikuti pattern project: Controller -> Service -> Repository -> Model.

================================================================================
1. RINGKASAN STATUS
================================================================================

Core aplikasi sudah terbentuk:
- Auth login/logout.
- Role Admin, HRD, Employee.
- Admin master data utama.
- Employee check-in/check-out.
- Employee history, calendar, leave request, correction request, profile.
- HRD dashboard, monitoring, approval, report CSV.

Namun beberapa business rule penting belum sempurna:
- Outside radius belum wajib memilih work mode.
- Status telat belum otomatis menjadi late.
- Early leave belum dihitung pada check-out normal.
- HRD belum check-in belum mengecualikan holiday, day off, dan approved leave.
- Leave approval belum membuat leave_request_details dan belum mengurangi leave_balances.
- Attachment leave/correction belum tersedia di form Employee.
- Attendance photo belum tampil di HRD detail/outside radius review.
- Activity logs dan approvals table belum digunakan konsisten.
- Report HRD belum lengkap sesuai BRD.

Target revisi utama:
- Membuat flow absensi lebih akurat.
- Membuat approval leave/correction lebih lengkap.
- Membuat HRD bisa memonitor dengan data yang benar.
- Membuat audit trail minimal tersedia.
- Menyiapkan fitur agar siap QA manual browser.

3. REVISION DETAIL - ATTENDANCE OUTSIDE RADIUS
================================================================================

MASALAH
-------
Saat check-in/check-out di luar radius kantor, sistem sudah mewajibkan note, tetapi work_mode_id belum wajib. Requirement bisnis menyatakan jika GPS di luar radius 100 meter, karyawan tetap boleh absen tetapi harus memberikan konteks: WFH, dinas luar, meeting luar, atau client visit.

FLOW YANG HARUS DIBUAT
----------------------
Jika user absen di dalam radius:
- check_in_is_inside_radius = true atau check_out_is_inside_radius = true.
- work_mode_id boleh default ke reference WORK_MODE.office.
- note boleh opsional.
- is_need_approval = false, kecuali ada rule lain.

Jika user absen di luar radius:
- check_in_is_inside_radius = false atau check_out_is_inside_radius = false.
- work_mode_id wajib diisi.
- note wajib diisi.
- work_mode_id tidak boleh office jika outside radius, kecuali bisnis memang mengizinkan.
- is_need_approval = true.
- attendance tetap tersimpan.
- HRD bisa melihat data ini di menu outside radius.

VALIDASI BACKEND
----------------
Pada check-in dan check-out:
- latitude wajib.
- longitude wajib.
- photo_data wajib.
- Jika distance_meter > radius_meter:
  - work_mode_id required.
  - note required.
  - work_mode_id harus termasuk group WORK_MODE.
  - disarankan work_mode_id bukan office.

VALIDASI FRONTEND
-----------------
- Saat GPS terdeteksi di luar radius, tampilkan alert: "Anda berada di luar radius kantor. Pilih mode kerja dan isi keterangan."
- Field work mode ditampilkan/wajib.
- Field note ditampilkan/wajib.
- Tombol submit disabled sampai work mode dan note terisi.

ACCEPTANCE CRITERIA
-------------------
- User di dalam radius bisa absen tanpa memilih work mode manual.
- User di luar radius tidak bisa submit tanpa work mode.
- User di luar radius tidak bisa submit tanpa note.
- Attendance outside radius tersimpan dengan is_need_approval = true.
- HRD bisa melihat work mode dan note pada detail attendance.

4. REVISION DETAIL - STATUS LATE OTOMATIS
================================================================================

MASALAH
-------
Saat check-in, late_minutes sudah dihitung, tetapi status attendance masih bisa tetap present. Ini membuat report status tidak akurat.

FLOW YANG HARUS DIBUAT
----------------------
Gunakan shift.start_time dan shift.late_tolerance_minutes.

Rule:
- Jika check_in_at <= shift_start + tolerance:
  - status_id = ATTENDANCE_STATUS.present
  - late_minutes = 0
- Jika check_in_at > shift_start + tolerance:
  - status_id = ATTENDANCE_STATUS.late
  - late_minutes = selisih menit dari batas toleransi atau dari shift_start, pilih satu rule dan konsisten.

REKOMENDASI PERHITUNGAN
-----------------------
Agar lebih adil, late_minutes dihitung setelah toleransi.
Contoh:
- Shift masuk 08:00.
- Tolerance 15 menit.
- Check-in 08:20.
- late_minutes = 5.
- status = late.

Jika perusahaan ingin menghitung telat dari jam masuk asli:
- Check-in 08:20.
- late_minutes = 20.

Pilih satu dan dokumentasikan di service.

ACCEPTANCE CRITERIA
-------------------
- Check-in sebelum batas toleransi menghasilkan status present.
- Check-in setelah batas toleransi menghasilkan status late.
- HRD late page konsisten dengan status late atau late_minutes > 0.
- Daily/monthly report menghitung late dari data yang sama.

5. REVISION DETAIL - EARLY LEAVE MINUTES CHECK-OUT
================================================================================

MASALAH
-------
Saat employee check-out normal, early_leave_minutes masih 0 walaupun pulang sebelum shift end.

FLOW YANG HARUS DIBUAT
----------------------
Pada check-out:
- Ambil shift.end_time.
- Bandingkan check_out_at dengan jadwal pulang.
- Jika check_out_at < shift_end_datetime:
  - early_leave_minutes = selisih menit.
- Jika check_out_at >= shift_end_datetime:
  - early_leave_minutes = 0.

OVERNIGHT SHIFT
---------------
Jika shifts.is_overnight = true:
- shift_end_datetime harus dianggap tanggal berikutnya.
- Contoh shift 22:00 - 06:00.
- Attendance date 2026-06-09.
- shift_end_datetime = 2026-06-10 06:00:00.

ACCEPTANCE CRITERIA
-------------------
- Check-out sebelum jam pulang menghasilkan early_leave_minutes > 0.
- Check-out setelah jam pulang menghasilkan early_leave_minutes = 0.
- Shift overnight tidak menghasilkan angka negatif.
- Attendance correction approval juga memakai perhitungan yang sama.

6. REVISION DETAIL - HOLIDAY, DAY OFF, DAN APPROVED LEAVE PADA HRD BELUM CHECK-IN
================================================================================

MASALAH
-------
Menu HRD belum check-in menampilkan employee aktif yang belum punya attendance pada tanggal terpilih, tetapi belum mengecualikan employee yang memang tidak wajib hadir.

FLOW YANG HARUS DIBUAT
----------------------
Employee tidak boleh masuk daftar belum check-in jika:
- Tanggal tersebut ada di holidays.
- Employee punya employee_schedules pada tanggal tersebut dengan is_day_off = true.
- Employee memiliki approved leave/sick/permission pada tanggal tersebut.

DETAIL RULE
-----------
1. Holiday:
   - Cek table holidays berdasarkan holiday_date.
   - Jika holiday aktif, default semua employee tidak wajib check-in.
   - Jika nanti ada jadwal khusus masuk saat holiday, bisa dikembangkan later.

2. Day off:
   - Cek employee_schedules.employee_id dan schedule_date.
   - Jika is_day_off = true, exclude dari belum check-in.

3. Approved leave:
   - Cek leave_request_details jika sudah dibuat.
   - Jika belum ada detail, cek leave_requests dengan start_date <= tanggal <= end_date dan status approved.
   - Exclude dari belum check-in.

ACCEPTANCE CRITERIA
-------------------
- Employee yang cuti approved tidak muncul sebagai belum check-in.
- Employee yang sakit approved tidak muncul sebagai belum check-in.
- Employee yang izin approved tidak muncul sebagai belum check-in.
- Employee yang day off tidak muncul sebagai belum check-in.
- Pada hari libur nasional/perusahaan, employee tidak muncul sebagai belum check-in.

7. REVISION DETAIL - LEAVE REQUEST APPROVAL
================================================================================

MASALAH
-------
Approval leave sudah mengubah status dan membuat/update attendance, tetapi belum membuat leave_request_details, belum mengurangi leave_balances, belum validasi kuota, dan belum validasi overlap dengan pengajuan lain.

FLOW SUBMIT LEAVE OLEH EMPLOYEE
-------------------------------
1. Employee membuka form leave request.
2. Employee memilih leave_type:
   - annual_leave
   - sick_leave
   - permission
3. Employee mengisi start_date, end_date, reason.
4. Employee upload attachment jika ada atau jika wajib.
5. Backend validasi:
   - start_date <= end_date.
   - Tidak overlap dengan leave request pending/approved lain.
   - Tidak overlap dengan attendance correction pending untuk tanggal yang sama jika relevan.
   - Jika annual_leave, cek leave_balances cukup.
6. Simpan leave_requests status pending.
7. Simpan attachment_id jika ada.
8. Simpan activity log.

FLOW APPROVE LEAVE OLEH HRD
---------------------------
1. HRD membuka detail leave request.
2. HRD klik approve.
3. Backend validasi ulang:
   - Status masih pending.
   - Tidak overlap dengan request approved lain.
   - Jika annual_leave, saldo masih cukup.
4. Update leave_requests:
   - status_id = approved.
   - approved_by = HRD user id.
   - approved_at = now.
   - approval_note = input HRD.
5. Buat leave_request_details per tanggal dari start_date sampai end_date.
6. Buat/update attendances per tanggal:
   - status_id sesuai leave type: leave/sick/permission.
   - employee_id sesuai request.
   - shift_id fallback dari employee shift.
   - work_location_id fallback dari employee work location.
   - attendance_date = tanggal detail.
7. Jika annual_leave:
   - update leave_balances.used_quota.
   - update leave_balances.remaining_quota.
8. Insert approvals table jika sudah mulai dipakai.
9. Insert activity_logs.

FLOW REJECT LEAVE OLEH HRD
--------------------------
1. HRD isi rejected_reason.
2. Update leave_requests:
   - status_id = rejected.
   - approved_by = HRD user id.
   - approved_at = now.
   - rejected_reason = input HRD.
3. Jangan buat leave_request_details.
4. Jangan buat/update attendances.
5. Jangan kurangi saldo cuti.
6. Insert activity_logs.

FLOW CANCEL LEAVE OLEH EMPLOYEE
-------------------------------
Saat ini cancel pending menghapus row. Ubah agar lebih aman:
- Jangan delete row.
- Ubah status_id = cancelled.
- Simpan updated_by.
- Jika memungkinkan tambah cancelled_at/cancelled_by pada migration later.
- Insert activity_logs.

ACCEPTANCE CRITERIA
-------------------
- Leave approved membuat leave_request_details sesuai jumlah tanggal.
- Leave approved membuat/update attendance status leave/sick/permission.
- Annual leave approved mengurangi leave balance.
- Leave rejected tidak mengubah attendance dan saldo.
- Leave cancelled tetap muncul di history dengan status cancelled.
- Overlap leave pending/approved ditolak.

8. REVISION DETAIL - LEAVE BALANCE
================================================================================

MASALAH
-------
leave_balances sudah ada dan HRD bisa read-only, tetapi belum terhubung ke annual leave.

FLOW YANG HARUS DIBUAT
----------------------
1. Saat employee submit annual_leave:
   - Cek leave_balances untuk employee dan year.
   - Jika remaining_quota < total_days, tolak request.

2. Saat HRD approve annual_leave:
   - Cek ulang remaining_quota.
   - used_quota += total_days.
   - remaining_quota = total_quota - used_quota.

3. Saat HRD reject annual_leave:
   - Tidak mengubah saldo.

4. Saat request cancelled sebelum approved:
   - Tidak mengubah saldo.

5. Jika nanti approved leave dibatalkan:
   - Kembalikan saldo. Ini bisa later.

REKOMENDASI UI
--------------
- HRD/Admin perlu halaman manage leave balance.
- Minimal untuk MVP: tambah action edit quota pada HRD leave balances.
- Later: import saldo cuti via CSV/Excel.

ACCEPTANCE CRITERIA
-------------------
- Employee tidak bisa submit cuti tahunan lebih dari sisa kuota.
- HRD tidak bisa approve jika saldo sudah tidak cukup.
- Saldo berubah setelah approve.
- Report dapat menampilkan total cuti terpakai.

================================================================================

MASALAH
-------
Correction sudah ada dan HRD approve bisa create/update attendance, tetapi attachment belum ada, validasi overlap/duplikasi belum ada, cancel masih delete row, dan approval table belum dipakai.

FLOW SUBMIT CORRECTION
----------------------
1. Employee memilih correction_date.
2. Employee mengisi requested_check_in_at dan/atau requested_check_out_at.
3. Employee mengisi reason.
4. Employee upload attachment optional.
5. Backend validasi:
   - Minimal salah satu dari requested_check_in_at/requested_check_out_at harus diisi.

7. Insert activity log.

FLOW APPROVE CORRECTION
-----------------------
1. HRD approve correction pending.
2. Backend create/update attendance pada correction_date.
3. Hitung ulang:
   - late_minutes.
   - total_work_minutes.
   - early_leave_minutes.
   - status_id present/late/incomplete sesuai data.
4. Buat attendance_logs action update_by_hrd.
5. Update correction status approved.
6. Insert approvals jika sudah dipakai.
7. Insert activity_logs.

FLOW REJECT CORRECTION
----------------------
1. HRD isi rejected_reason.
2. Update correction status rejected.
3. Jangan mengubah attendance.
4. Insert activity_logs.

FLOW CANCEL CORRECTION
----------------------
- Jangan delete row.
- Ubah status_id = cancelled.
- Insert activity_logs.

ACCEPTANCE CRITERIA
-------------------
- Tidak bisa membuat 2 correction pending untuk tanggal sama.
- Approve correction menghitung ulang late/work/early leave.
- Reject correction tidak mengubah attendance.
- Cancel correction tetap muncul di history.
- HRD bisa melihat attachment correction.

18. REVISION DETAIL - APPROVALS TABLE
================================================================================

MASALAH
-------
approvals table/model ada, tetapi approval saat ini langsung update tabel target.

REKOMENDASI IMPLEMENTASI
------------------------
Gunakan approvals sebagai audit terpusat, bukan pengganti status di tabel target.

Saat HRD approve/reject leave:
- Update leave_requests seperti sekarang.
- Insert approvals:
  - approvable_type = leave_request.
  - approvable_id = leave_requests.id.
  - requested_by = employee user_id jika tersedia.
  - approved_by = HRD user_id.
  - status_id = approved/rejected.
  - note = approval_note/rejected_reason.
  - approved_at = now.

Saat HRD approve/reject correction:
- approvable_type = attendance_correction.

Saat HRD review outside radius:

19. REVISION DETAIL - OUTSIDE RADIUS APPROVAL/REJECT FLOW
================================================================================

MASALAH
-------
Reject outside radius belum punya efek/status jelas. Saat ini reject hanya seperti memberi catatan.

FLOW YANG HARUS DIPUTUSKAN
--------------------------
Pilih salah satu pendekatan berikut:

OPSI A - SEDERHANA UNTUK MVP
- Attendance tetap ada.
- HRD review outside radius dengan approve/reject.
- Jika approve:
  - is_need_approval = false.
  - approval_note terisi.
- Jika reject:
  - is_need_approval = false.
  - status_id = absent atau pending_approval/rejected equivalent jika ada reference.
  - approval_note/rejected_reason terisi.

OPSI B - LEBIH AMAN
- Tambah review_status_id pada attendances atau gunakan approvals table.
- Attendance status tetap present/late.
- Approval status terpisah: pending, approved, rejected.
- Report bisa membedakan hadir tetapi lokasi ditolak.

REKOMENDASI
-----------
Gunakan Opsi B jika tidak ingin mengubah status hadir langsung.
Untuk MVP cepat, minimal gunakan approvals table agar reject tercatat jelas.

ACCEPTANCE CRITERIA
-------------------
- HRD bisa approve outside radius.
- HRD bisa reject outside radius.
- Hasil approve/reject terlihat di attendance detail.
- Report dapat membedakan outside radius pending/approved/rejected.
